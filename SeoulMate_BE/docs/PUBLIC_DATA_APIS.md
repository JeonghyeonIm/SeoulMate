# SeoulMate_BE Public Data APIs

## 개요

이 문서는 `SeoulMate_BE`에서 사용할 서울 공공데이터 API 후보를 정리한 문서입니다.
기준일은 `2026-05-06`이며, 현재 프로젝트의 DB 구조와 추천 기능 요구사항을 기준으로 우선순위를 나눴습니다.

현재 프로젝트에서 공공데이터는 주로 아래 목적에 사용됩니다.

- `public_data` 테이블 초기 적재
- 추천 후보 장소 수집
- 추천 점수 계산 보강
- 사용자 요청 조건과 장소 메타데이터 매칭

## 우선순위 요약

### 1순위

- 서울시 문화행사 정보 / 문화공간 정보
- 서울시 실시간 도시데이터 또는 실시간 인구데이터
- 서울시 식품위생업소 현황

### 2순위

- 서울시 공공서비스예약 정보
- 서울교통공사 지하철역 정보 계열
- 서울시 실시간 자치구별 대기환경 현황

## API별 정리

### 1. 서울시 문화행사 정보 / 문화공간 정보

- 목적:
  - 문화, 관광, 전시, 공연, 실내 활동 장소 후보 확보
  - 지역 기반 추천, 기간 기반 추천에 활용
- 활용 테이블:
  - `public_data`
- 주요 적재 컬럼:
  - `title`
  - `category`
  - `region`
  - `address`
  - `source`
  - `source_url`
  - `metadata`
- 추천 활용 포인트:
  - 문화/관광 카테고리 추천
  - 날짜가 있는 이벤트성 추천
  - 사용자 선호 카테고리와 직접 매칭
- 비고:
  - 문화행사는 기간 정보가 있어 `metadata`에 시작일/종료일을 같이 보관하는 것이 좋음
- 공식 참고:
  - https://data.seoul.go.kr/dataList/17/literacyView.do

### 2. 서울시 실시간 도시데이터

- 목적:
  - 추천 점수에 혼잡도, 이동 여건, 주변 실시간 상태 반영
- 활용 테이블:
  - 직접 `public_data`에 영구 저장하기보다
  - 추천 계산 시 조회하거나 별도 캐시 테이블로 저장하는 방향 권장
- 주요 활용 항목:
  - 실시간 인구
  - 교통/도로
  - 문화행사 현황
  - 날씨/환경
- 추천 활용 포인트:
  - 혼잡도 낮은 장소 우선 추천
  - 특정 시간대 회피 추천
  - 안전하거나 덜 붐비는 코스 구성
- 비고:
  - 서울 주요 120~121개 장소 중심 데이터라서 개별 맛집/카페와 1:1 매핑되진 않음
  - 따라서 `place -> 대표 실시간 지역 코드` 매핑 전략 필요
- 공식 참고:
  - https://data.seoul.go.kr/dataList/OA-22714/A/1/datasetView.do

### 3. 서울시 실시간 인구데이터

- 목적:
  - 실시간 도시데이터보다 단순한 혼잡도 반영
- 활용 테이블:
  - 캐시성 저장 또는 추천 시 실시간 조회
- 추천 활용 포인트:
  - 혼잡도 점수 반영
  - 사용자 요청이 "한적한 곳"일 때 필수 보조 데이터
- 비고:
  - 실시간 도시데이터를 쓰면 일부 대체 가능
  - 둘 중 하나만 먼저 붙인다면 일반적으로는 `실시간 도시데이터`가 더 범용적
- 공식 참고:
  - https://data.seoul.go.kr/dataList/OA-21778/A/1/datasetView.do

### 4. 서울시 식품위생업소 현황

- 목적:
  - 맛집, 카페 후보군 확보
- 활용 테이블:
  - `public_data`
- 주요 적재 컬럼:
  - `title`
  - `category`
  - `region`
  - `address`
  - `source_record_id`
  - `metadata`
- 추천 활용 포인트:
  - 맛집/카페 중심 추천
  - 지역 기반 식음료 장소 후보 확장
- 비고:
  - 추천 품질을 바로 보장하는 데이터는 아니고, 후보 모수 확보용 데이터에 가까움
  - 업종코드, 영업상태, 행정동/법정동 정보는 `metadata`에 보관 추천
- 공식 참고:
  - https://data.seoul.go.kr/dataList/datasetList.do
  - 검색어: `서울시 식품위생업소 현황`

### 5. 서울시 공공서비스예약 정보

- 목적:
  - 예약 가능한 문화/체육/시설 프로그램과 연계
- 활용 테이블:
  - `public_data`
  - 필요 시 별도 예약 메타 테이블
- 주요 적재 컬럼:
  - `title`
  - `category`
  - `region`
  - `source_url`
  - `metadata`
- 추천 활용 포인트:
  - 추천 후 실제 예약 행동까지 이어지는 UX 구성
  - 행사성/체험형 추천 강화
- 비고:
  - 예약 링크나 서비스 ID는 `metadata`에 넣는 게 현실적
- 공식 참고:
  - https://www.data.go.kr/data/15134352/openapi.do
  - 원문 연결: 서울 열린데이터광장 `OA-2271`

### 6. 서울교통공사 지하철역 정보 계열

- 목적:
  - 장소 접근성, 이동 편의성 계산
- 활용 테이블:
  - 직접 적재 시 별도 교통 메타 테이블이 더 적절하지만
  - 초기에는 장소 추천 점수 계산용 보조 데이터로 사용하는 것이 현실적
- 추천 활용 포인트:
  - 가까운 역 존재 여부
  - 이동 부담 감소
  - 대중교통 친화 코스 추천
- 비고:
  - 처음부터 모든 교통 데이터를 정규화하기보다, 역명/좌표/노선 정도만 보조적으로 쓰는 것이 낫습니다.
- 공식 참고:
  - https://data.seoul.go.kr/dataList/32/literacyView.do

### 7. 서울시 실시간 자치구별 대기환경 현황

- 목적:
  - 야외 활동 추천 품질 보강
- 활용 테이블:
  - 실시간성 보조 데이터 또는 캐시 테이블
- 추천 활용 포인트:
  - 산책, 공원, 야외 관광 추천 시 대기질 반영
  - 미세먼지 높은 날 실내 추천 강화
- 비고:
  - `region` 기준 조인/매핑이 쉬운 편
- 공식 참고:
  - https://data.seoul.go.kr/dataList/OA-1200/A/1/datasetView.do

## 현재 프로젝트 기준 추천 조합

### 최소 구성

아래 3개만 먼저 붙여도 1차 서비스는 가능합니다.

1. 문화행사/문화공간 정보
2. 실시간 도시데이터
3. 식품위생업소 현황

### 추천 확장 구성

아래까지 붙이면 추천 품질이 더 좋아집니다.

1. 문화행사/문화공간 정보
2. 실시간 도시데이터
3. 식품위생업소 현황
4. 공공서비스예약 정보
5. 지하철역 정보
6. 대기환경 현황

## DB 매핑 전략

### `public_data`에 바로 적재할 데이터

- 문화행사/문화공간 정보
- 식품위생업소 현황
- 공공서비스예약 정보

### 추천 계산용 보조 데이터

- 실시간 도시데이터
- 실시간 인구데이터
- 지하철역 정보
- 대기환경 현황

## 수집 전략 권장안

### 1차 적재 배치

- 주기:
  - 하루 1회 또는 몇 시간 단위
- 대상:
  - 문화행사/문화공간
  - 식품위생업소
  - 예약 정보

### 실시간/준실시간 조회

- 주기:
  - 추천 요청 시점 조회 또는 짧은 TTL 캐시
- 대상:
  - 실시간 도시데이터
  - 실시간 인구데이터
  - 대기환경

### 운영 메모

- 실시간 데이터는 추천 요청마다 외부 API를 직접 때리면 느릴 수 있으니 캐시 권장
- `public_data_sync_runs` 테이블에 적재 성공/실패 이력을 남기는 방식이 좋음
- 원천 API별로 `source`, `source_dataset`, `source_record_id`를 일관되게 저장해야 중복 제어가 쉬움

## 실제 구현 순서 제안

1. `문화행사/문화공간 정보`를 `syncPublicData.ts`에 먼저 연결
2. `식품위생업소 현황`을 추가해 `맛집/카페` 후보군 확장
3. `실시간 도시데이터`를 추천 점수 계산에 연결
4. 필요 시 `공공서비스예약 정보`를 붙여 행동 전환 강화
5. 이후 `지하철역 정보`, `대기환경`으로 점수 정교화
